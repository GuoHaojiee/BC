## 一、这个项目在干什么？

### 背景：什么是介数中心性（Betweenness Centrality，BC）？

想象你是一张地图上的交叉路口。如果城市里大多数人从 A 走到 B 都必须经过你，那你就是一个"关键节点"——去掉你，很多路就断了。

**介数中心性**就是量化这种"关键程度"的指标：

```
BC(v) = 所有顶点对 (s, t) 中，经过 v 的最短路径数
        ─────────────────────────────────────────────
              s 到 t 的最短路径总数（求和）
```

BC 值越高，说明这个节点在图中越"关键"——它是信息传播、网络流量、社会影响力的枢纽。

**应用场景举例：**
- 社交网络：找到影响力最大的 KOL
- 交通网络：找到最易引发拥堵的路口
- 网络安全：找到最值得攻击/保护的服务器节点
- 生物网络：找到蛋白质相互作用中的关键蛋白

### 问题的规模

对一个有 n 个顶点、m 条边的图，暴力算法需要 **O(n³)** 时间。

对于本项目测试用的 RMAT 图（scale=12，即 n=4096，m≈65536）：
- 单核暴力：`4096³ ≈ 689 亿次运算`，需要几小时
- 用 Brandes 算法 + GPU 并行：秒级完成

## 二、Brandes 算法：把 O(n³) 降到 O(n·(n+m))

### 核心思想

Brandes（2001）的天才发现：不需要枚举所有路径，只需要对每个源节点 s 跑一次 BFS，就能高效地计算所有经过每个节点的"依存值"。

整个算法分两个阶段：

```
对图中每个源节点 s：
    ┌─ 阶段一：正向 BFS ─────────────────────────────────┐
    │  从 s 出发做广度优先搜索                            │
    │  计算 d[v]    = s 到 v 的最短距离                   │
    │  计算 σ[v]    = s 到 v 的最短路径数量               │
    │  记录 stack   = 顶点按 BFS 访问顺序的序列           │
    └─────────────────────────────────────────────────────┘
           ↓ 反转
    ┌─ 阶段二：反向传播（Back Propagation）──────────────┐
    │  从 stack 末尾（最深层）往回处理                    │
    │  计算 δ[v]（依存值）：                              │
    │    对每个前驱 u（d[u] == d[v]-1 的邻居）：          │
    │      δ[u] += σ[u]/σ[v] * (1 + δ[v])               │
    │  BC[v] += δ[v]  （v 不是源节点时）                 │
    └─────────────────────────────────────────────────────┘
```

对所有 n 个源节点做完后，`BC[v] / 2` 就是顶点 v 的最终介数中心性（除以 2 是因为无向图每条路径被正反各算了一次）。

### 为什么这个公式是对的？

**直觉解释**：`δ_s(v)` 表达的是"从 s 出发，经过 v 的最短路径，最终能到达多少个目标节点，并且路径上 v 的贡献是多少"。

把所有源节点的 `δ_s(v)` 加起来，就等价于枚举了所有经过 v 的最短路径对——但避免了显式枚举。

## 三、并行化策略

### 为什么天然适合并行？

关键观察：**不同源节点 s 的 BFS 计算完全独立，互不干扰。**

```
n 个源节点的计算 = n 个独立的 BFS + 反向传播

并行策略：让不同的计算单元各自负责不同的源节点
          ├── MPI：不同进程负责不同的源节点子集
          └── GPU：同一进程内，不同 CUDA block 并行处理不同源节点
```

### 本项目的两层并行

```
┌─────────────────────────────────────────────────────────┐
│                   MPI 进程级并行                         │
│                                                          │
│  进程 0（GPU 0）         进程 1（GPU 1）                 │
│  负责顶点 0~2047         负责顶点 2048~4095              │
│  以这些顶点为源节点      以这些顶点为源节点              │
│  跑 GPU 计算             跑 GPU 计算                     │
│         │                        │                       │
│         └──── MPI_Allreduce ─────┘                       │
│               汇总所有贡献，得到完整 BC                  │
└─────────────────────────────────────────────────────────┘

单个 GPU 内部：
┌─────────────────────────────────────────────────────────┐
│  block 0      block 1      block 2  ...  block 63        │
│  处理源节点0  处理源节点1  处理源节点2 ... 处理源节点63  │
│  256线程并行  256线程并行  256线程并行    256线程并行    │
│  处理出边     处理出边     处理出边       处理出边       │
└─────────────────────────────────────────────────────────┘
```

## 四、代码详解

### 4.1 `brandes_gpu.cu` — CUDA 核心实现

#### 内存布局

GPU 同时处理 `BATCH_SIZE=64` 个源节点，所有工作数组按源节点索引展开：

```
d[src_idx * n + v]     = 源节点 src_idx 到顶点 v 的 BFS 距离
sigma[src_idx * n + v] = 源节点 src_idx 到顶点 v 的最短路径数
delta[src_idx * n + v] = 顶点 v 对源节点 src_idx 的依存值
stack[src_idx * n + k] = 第 k 个被 BFS 访问到的顶点（顺序记录）
```

每个 block 只访问自己对应 `src_idx` 的那一行，**block 之间完全不共享工作内存**，避免了任何跨 block 同步。

#### `bfs_kernel`：正向 BFS

```
一个 block（256线程）处理一个源节点 src

初始化：d[src]=0, sigma[src]=1, 其余=-1/0
        ↓
循环（每次处理一层）：
        ↓
┌── 第一遍：发现新顶点 ───────────────────────────────────┐
│  256线程分摊当前层的所有顶点                            │
│  thread_i 处理 stack[lvl_start + i, i+256, i+512, ...] │
│                                                         │
│  for 每个当前层顶点 v：                                 │
│    for 每条出边 (v, w)：                                │
│      if atomicCAS(d[w], -1, d[v]+1) == -1:            │
│         ← 说明 w 第一次被发现                           │
│         pos = atomicAdd(stack_size, 1)                  │
│         stack[pos] = w   ← 加入队列                    │
└─────────────────────────────────────────────────────────┘
        ↓  __syncthreads()  (确保所有 d[w] 已写入)
┌── 第二遍：累加 sigma ───────────────────────────────────┐
│  for 每个当前层顶点 v：                                 │
│    for 每条出边 (v, w)：                                │
│      if d[w] == d[v]+1:   ← w 是下一层的顶点           │
│        atomicAdd(sigma[w], sigma[v])                    │
│          ← 多个前驱 v 并发原子加，正确累积              │
└─────────────────────────────────────────────────────────┘
        ↓  __syncthreads()
   lvl_start = lvl_end
   lvl_end   = stack_size    ← 推进到下一层
        ↓
重复，直到 lvl_start == lvl_end（BFS 结束）
```

**为什么要两遍？** 第一遍的 `atomicCAS` 保证每个顶点只入队一次，但多个线程可能同时把不同的 v 的 sigma 贡献给同一个 w。必须等第一遍（d[w] 全部设好）完成后，第二遍才能正确判断"w 是不是下一层顶点"。`__syncthreads()` 是关键屏障。

#### `back_prop_kernel`：反向传播

```
反向传播从最深层（max_lvl）处理到第 1 层：

for lvl = max_lvl 到 1：
    ┌── 传播 delta ──────────────────────────────────────┐
    │  256线程扫描全部 n 个顶点                          │
    │  for 每个满足 d[v]==lvl 的顶点 v：                 │
    │    coeff = (1 + delta[v]) / sigma[v]               │
    │    for 每条边 (v, u)：                             │
    │      if d[u] == lvl-1:   ← u 是 v 的前驱          │
    │        atomicAdd(delta[u], sigma[u] * coeff)       │
    └────────────────────────────────────────────────────┘
        ↓  __syncthreads()  (当前层所有 delta 传播完毕)
    ┌── 汇入 BC ─────────────────────────────────────────┐
    │  for 每个满足 d[v]==lvl 且 v!=src 的顶点：        │
    │    atomicAdd(bc_out[v], delta[v])                  │
    │      ← 跨 block（源节点）并发写，必须原子          │
    └────────────────────────────────────────────────────┘
        ↓  __syncthreads()
```

**为什么 `bc_out` 也需要 `atomicAdd`？** `bc_out` 是所有 block 共享的全局数组。64 个 block 可能同时对同一个顶点 v 执行 `bc_out[v] += delta[v]`（来自 64 个不同源节点），必须原子操作。P100（sm_60）原生支持 `atomicAdd(double*, double)`。

#### `computeBC_gpu`：批次调度

```
上传图 CSR 到 GPU（常驻）
初始化 bc_out = 0（跨批次保留，累加）

for batch_start = 0, 64, 128, ..., n_src：
    上传本批 64 个源节点编号
    launch bfs_kernel<<<64 blocks, 256 threads>>>
    launch back_prop_kernel<<<64 blocks, 256 threads>>>

cudaDeviceSynchronize()
下载 bc_out 到主机
释放所有显存
```

### 4.2 `solution_mpi.cpp` — `run()` 函数实现

#### 第一步：组装完整图（MPI 通信）

**问题**：图是分布式存储的，进程 0 只有顶点 0\~2047 的边数据，GPU BFS 需要完整图。

**解决方案**：用 `MPI_Allgatherv` 让每个进程都拿到完整图。

```
每个进程持有（分布式存储）：
  rowsIndices[0..local_n]  ← 本地 CSR 偏移，从 0 开始
  endV[0..local_m-1]       ← 出边目标顶点（全局编号）

通信后每个进程得到：
  full_offset[0..n]        ← 完整图 CSR 偏移
  full_dest[0..m-1]        ← 完整图出边目标
```

**关键：偏移的修正**

本地 `rowsIndices` 从 0 开始（相对偏移），但全局 `full_offset` 需要绝对偏移：

```
full_offset[全局顶点 v] = 进程 p 的边数前缀和 + v 的本地偏移

具体实现：
  endv_disp[p] = 进程 0..p-1 的边数之和（前缀和）
  full_offset[g_start_p + l] = endv_disp[p] + rowsIndices_p[l]
```

**通信量分析（scale=12，2 进程）：**

| 通信操作 | 数据量 |
|---------|-------|
| Allgather local_m | 2 × 4B = 8B |
| Allgatherv endV | 131072 × 4B ≈ 512KB |
| Allgatherv rowsIndices | 4096 × 4B = 16KB |

总计不到 1MB，开销极小。

#### 第二步：GPU 计算（部分 BC）

```
进程 0 的源节点：[0, 1, 2, ..., 2047]（全局编号）
进程 1 的源节点：[2048, 2049, ..., 4095]（全局编号）

GPU 输出 bc_partial[n]：
  bc_partial[v] = 仅来自本进程源节点的 BC 贡献（不完整！）
```

#### 第三步：MPI_Allreduce 汇总（关键步骤！）

```
进程 0 的 bc_partial：[来自源0..2047 对各顶点的贡献]
进程 1 的 bc_partial：[来自源2048..4095 对各顶点的贡献]
                    ↓ MPI_Allreduce(SUM)
所有进程的 bc_partial：[完整 BC，来自全部 n 个源节点]
```

**为什么不能省略这一步？** 顶点 v 的 BC 定义是对**所有** n 个源节点求和。每个进程只算了 local_n 个源节点的贡献，是不完整的。Allreduce 把各进程的部分结果加在一起，才得到正确答案。

#### 第四步：提取本地结果

```
result[i] = bc_partial[sources[i]]  （本地编号 i → 全局编号 → 取值）
result[i] /= 2.0                    （无向图修正）
```
